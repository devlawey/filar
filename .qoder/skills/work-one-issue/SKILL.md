# work-one-issue

Обработать одну issue по флоу «одна issue = одна ветка = один PR» и остановиться
на ревью. Универсально для любой issue: всё специфичное берётся из тела самой issue;
инварианты проекта — из AGENTS.md.

**Аргумент:** номер issue (через $ARGUMENTS). Если не передан — взять из целевого
milestone самую приоритетную ещё открытую issue, не находящуюся в работе, и сообщить
пользователю, какая выбрана.

## Процедура

Соблюдай правила из AGENTS.md в корне репозитория. Работай строго над ОДНОЙ issue
за раз и останавливайся на ревью.

1. Прочитай тело указанной issue ($ARGUMENTS) полностью. Оно — единственный источник
   правды: что менять, в каких файлах и каковы критерии готовности. Не придумывай
   требований сверх описанных; если в issue чего-то критично не хватает или оно
   противоречиво — не угадывай, остановись и спроси пользователя.

2. Создай ветку от актуального `main`. Имя: `fix/<номер>-<краткий-слаг>`, где слаг —
   2-4 слова из заголовка issue в kebab-case. Префикс по характеру: `fix/`, `feat/`
   или `refactor/`.

3. Реализуй изменения строго по телу issue. Соблюдай инварианты из AGENTS.md
   (zero-install на удалёнке, никаких команд без подтверждения, секреты только из env,
   изоляция за трейтами). Меняй только то, что относится к этой issue.

4. Если в критериях готовности есть тесты — добавь/обнови их. Прогони
   `cargo build --workspace` и `cargo test --workspace`: сборка и юнит-тесты должны
   быть зелёными. Тесты с `#[ignore]` (требуют docker-sshd) сам не запускай — перечисли
   их в описании PR как «требуют ручной проверки».

5. Обнови `PROGRESS.md`: кратко отметь что сделано, принятые решения и изменения
   публичных контрактов. Это входит в ТУ ЖЕ правку (тот же коммит/PR).

6. Сделай ОДИН логический коммит (код + PROGRESS.md). Сообщение в Conventional
   Commits: `<type>(<scope>): <описание>`.

7. Запушь ветку и открой Pull Request в `main`. В описании обязательно: строка
   `Closes #<номер>`; краткое summary; пометка про тесты (что прошло, что требует
   ручной проверки); что НЕ вошло в скоуп, если всплыло смежное.

8. ОСТАНОВИСЬ. Не мержи PR. Не бери следующую issue. Жди ревью пользователя и
   CodeRabbit.

## Ограничения
- Не трогай `main` напрямую и не делай force-push.
- Не закрывай и не редактируй другие issue.
- Один PR решает ровно одну issue.
