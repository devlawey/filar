# План модернизации TUI Filar: мышь, стриминг, современный минималистичный дизайн

> Документ-задание для агента (Qoder + GLM). Каждая задача самодостаточна, выполняется
> последовательно, содержит контекст, конкретные шаги и **Definition of Done (DoD)**.
> После каждой задачи проект обязан собираться (`cargo build`) и проходить
> `cargo clippy --all-targets -- -D warnings` и `cargo test`.

---

## Дизайн-философия

> **Перед началом каждой задачи прочитать `DESIGN_PHILOSOPHY.md`** — он лежит
> рядом с этим планом и является обязательным контекстом для всех решений.
> Краткая выжимка: минимум рамок (воздух, разделители `─`, gutter `│`);
> все цвета только из `Theme`; цвет = смысл; единственная анимация — спиннер;
> мышь дополняет клавиатуру, каждый клик имеет хоткей-эквивалент; полная
> обратная совместимость (хоткеи с русской раскладкой, `!`, Ctrl+T/P/C,
> секреты, сессии); Windows-first с ASCII-фоллбэками для conhost;
> anti-scope соблюдать строго.

event.rs, terminal.rs}`, `crates/agent/src/{glm.rs, agent.rs}`, `crates/core/src/chat.rs`.

## Задача 1. Модуль темы и рефакторинг рендера

**Контекст.** Сейчас `ui.rs` (~440 строк) — один файл с жёстко зашитыми цветами.
Прежде чем менять внешний вид и добавлять мышь, нужно навести порядок.

**Шаги:**

1. Создать `crates/tui/src/theme.rs` со структурой:
   ```rust
   pub struct Theme {
       pub bg: Color,            // фон (обычно Color::Reset — фон терминала)
       pub fg: Color,            // основной текст
       pub fg_dim: Color,        // вторичный текст (серый)
       pub fg_muted: Color,      // самый тусклый (подсказки, плейсхолдеры)
       pub accent: Color,        // главный акцент (пользователь, фокус) — Cyan
       pub success: Color,       // agent, approved — Green
       pub warning: Color,       // thinking, внимание — Yellow
       pub danger: Color,        // ошибки, деструктивные команды — Red
       pub surface: Color,       // фон "приподнятых" элементов (статус-бар, кнопки)
       pub selection_bg: Color,  // фон выделения текста мышью
   }
   impl Theme { pub fn default_dark() -> Self { ... } }
   ```
   Добавить методы-хелперы: `theme.user_style()`, `theme.agent_style()`,
   `theme.error_style()`, `theme.dim()`, `theme.muted()` и т.п.
2. Разбить `ui.rs` на модуль `crates/tui/src/ui/`:
   - `ui/mod.rs` — `pub fn render(...)` + layout;
   - `ui/theme.rs` — тема (перенести из п.1);
   - `ui/chat.rs` — рендер истории чата;
   - `ui/input.rs` — поле ввода / confirm / password;
   - `ui/bars.rs` — статус-бар и help-бар;
   - `ui/text.rs` — `strip_emoji`, `wrap_text` (перенести без изменений).
3. Заменить все прямые `Color::*` на обращения к `Theme`. Экземпляр темы хранить
   в `App` (поле `pub theme: Theme`).
4. Поведение и внешний вид на этом шаге **не меняются** (кроме того, что цвета
   теперь идут из одной точки).

**DoD:**
- `cargo build` и `cargo clippy -- -D warnings` без ошибок.
- В `crates/tui/src/ui/` нет ни одного литерала `Color::` вне `theme.rs`.
- Внешний вид приложения идентичен прежнему (ручная проверка).
- Существующие тесты проходят.

## Задача 2. Кэширование layout чата (фундамент для мыши и скролла)

**Контекст.** Сейчас `render_chat_history` каждый кадр (до 60 fps) заново
разворачивает все `ChatBlock` в строки и переносит их по ширине. Для мыши нужен
hit-testing: «в какой блок попал клик по координате (x, y)?» — значит, нужна
кэшированная карта строк.

**Шаги:**

1. Создать `crates/tui/src/ui/layout_cache.rs`:
   ```rust
   /// Одна отрендеренная строка чата с привязкой к исходному блоку.
   pub struct RenderedLine {
       pub line: ratatui::text::Line<'static>,
       pub block_index: Option<usize>, // индекс в app.messages; None = пустая строка-разделитель
       pub region: LineRegion,         // Header | Body | OutputToggle | ...
   }
   pub enum LineRegion { Header, Body, Output, OutputToggle, Spacer }

   pub struct ChatLayoutCache {
       pub lines: Vec<RenderedLine>,
       width: u16,           // ширина, для которой построен кэш
       message_count: usize, // messages.len() на момент построения
       last_block_rev: u64,  // ревизия последнего блока (для стриминга, задача 7)
   }
   ```
2. Метод `ChatLayoutCache::rebuild(&mut self, messages: &[ChatBlock], width: u16, theme: &Theme, collapsed: &HashSet<usize>)`
   переносит логику построения строк из `chat.rs`. Инвалидация: изменилась ширина,
   изменилось число сообщений или ревизия последнего блока.
3. В `App` добавить: `pub layout_cache: ChatLayoutCache` и
   `pub message_rev: u64` (инкрементировать при любой мутации `messages`; ввести
   приватный метод `push_message()` вместо прямых `self.messages.push(...)` —
   заменить все вхождения).
4. `render_chat_history` теперь: проверить/перестроить кэш → взять срез видимых
   строк по `scroll` → отрисовать. Лимит `MAX_LINES = 500` заменить на лимит
   в 2000 кэшированных строк (кэш делает это дёшево).
5. В `App` добавить `pub chat_area: Rect` — фактическая область чата на экране,
   заполняется при каждом рендере (нужно для hit-testing в задаче 3). Аналогично
   `pub input_area: Rect`, `pub confirm_button_areas: Vec<(Rect, bool)>`
   (заполним позже).

**DoD:**
- Внешний вид не изменился.
- При длинной истории (200+ сообщений) прокрутка не тормозит (перестройка кэша
  не происходит на каждом кадре — проверить логом/счётчиком в debug).
- Unit-тест: кэш инвалидируется при изменении ширины и добавлении сообщения,
  и НЕ перестраивается при повторном вызове с теми же параметрами.

## Задача 3. Инфраструктура мыши: захват, событие, скролл колесом

**Контекст.** Мышь не захватывается вовсе. Начинаем с самого ценного — скролла.

**Шаги:**

1. В `runner.rs::run()`: при инициализации добавить `EnableMouseCapture`,
   при завершении — `DisableMouseCapture` (в обоих путях выхода, включая ошибочный):
   ```rust
   crossterm::execute!(stdout, EnterAlternateScreen, EnableMouseCapture)...
   // teardown:
   crossterm::execute!(io::stdout(), DisableMouseCapture, LeaveAlternateScreen).ok();
   ```
2. В event loop `runner.rs` обработать `Event::Mouse(m)`:
   ```rust
   Some(Ok(Event::Mouse(m))) => { app.handle_mouse(m); needs_redraw = true; }
   ```
3. В `app.rs` добавить `pub fn handle_mouse(&mut self, m: MouseEvent)`.
   Первая функциональность — колесо в области чата (все режимы, кроме Interactive):
   - `MouseEventKind::ScrollUp` внутри `self.chat_area` → `scroll += 3`;
   - `ScrollDown` → `scroll = scroll.saturating_sub(3)`;
   - клампить `scroll` максимумом (`layout_cache.lines.len().saturating_sub(visible_height)`),
     чтобы нельзя было укрутить в пустоту. Тот же кламп применить к PgUp.
4. Индикатор «вы не внизу»: если `scroll > 0`, в правом нижнем углу области чата
   рисовать ненавязчивую метку `↓ N new` тусклым цветом (`theme.fg_muted`), где N —
   количество строк ниже вьюпорта. Клик по ней (обработаем в задаче 4) и `End` —
   сброс в низ. Добавить обработку `KeyCode::End` при пустом вводе → `scroll = 0`.

**Проверка на Windows Terminal и conhost обязательна** (проект Windows-first):
скролл колесом должен работать в обоих.

**DoD:**
- Колесо мыши прокручивает чат вверх/вниз в Normal/Thinking/Confirming.
- Скролл клампится, метка `↓ N new` появляется/исчезает корректно.
- PgUp/PgDn работают как раньше.
- Захват мыши корректно снимается при выходе (после закрытия приложения
  выделение текста в терминале ОС работает).

## Задача 4. Скроллбар и hit-testing кликов

**Шаги:**

1. Отрисовать вертикальный скроллбар по правому краю области чата через
   `ratatui::widgets::{Scrollbar, ScrollbarState, ScrollbarOrientation::VerticalRight}`.
   Стиль: тонкий, thumb — `theme.fg_dim`, track — невидимый или `theme.fg_muted`.

   сбрасывать на `Up`.
3. Общий hit-testing: приватный метод
   `fn hit_test(&self, col: u16, row: u16) -> HitZone`, где
   ```rust
   enum HitZone { Chat { line_idx: usize }, ChatEmpty, Scrollbar, Input, HelpBar, StatusBar, ConfirmButton(bool), Outside }
   ```
   `line_idx` вычисляется из `row`, `chat_area` и текущего `scroll` через

4. Клик в область ввода → перевод фокуса (пока фокус один, но: клик по колонке
   внутри введённого текста ставит `cursor_pos` в соответствующую позицию —
   учесть перенос строки так же, как в существующей математике курсора).
5. Клик по метке `↓ N new` → `scroll = 0`.

**DoD:**
- Скроллбар отображается при переполнении, перетаскивается мышью.
- Клик в поле ввода ставит курсор под клик (включая вторую строку wrap).
- Unit-тест на `hit_test`: подготовить фиктивные `Rect` и проверить попадания
  по зонам и вычисление `line_idx`.

## Задача 5. Кликабельное подтверждение команд (модальный диалог с кнопками)

**Контекст.** Сейчас подтверждение — текст в нижней панели, ответ только с
клавиатуры. Делаем аккуратный центрированный модал с кнопками.

**Шаги:**

1. В `ui/input.rs` (или новый `ui/confirm.rs`): в режиме `Confirming` рисовать
   поверх чата центрированный блок (ширина `min(70, area.width - 8)`, высота по
   контенту, `Clear` под ним):
   - заголовок: `⚠ Confirm command` (danger, bold) — или ` Confirm command `
     без символа, ASCII-безопасно;
   - объяснение (`fg`), при `destructive` — строка-предупреждение (danger);
   - сама команда: строка с префиксом `$ `, цвет `warning`, при длинной команде —
     wrap;
   - пустая строка;
   - кнопки: `[ Approve (a) ]` и `[ Deny (d) ]` — прямоугольники с фоном
     `theme.surface`, текст success/danger соответственно; между ними 3 пробела;
   - выбранная по умолчанию кнопка (Deny — безопасный дефолт) подсвечена
     инверсией; Tab/←/→ переключают выбор, Enter активирует выбранную.
2. Рамка модала — единственное место, где рамка уместна: `Borders::ALL`,
   скруглённые углы `BorderType::Rounded`, цвет `danger` при destructive,
   иначе `warning`.
3. Прямоугольники кнопок записывать в `app.confirm_button_areas: Vec<(Rect, bool)>`
   при рендере (bool = approve). В `handle_mouse`: `Down(Left)` по кнопке →
   `respond_to_confirmation(approve)`. Hover (`MouseEventKind::Moved`) —
   подсветка кнопки под курсором (хранить `hovered_button: Option<bool>`).
4. Клавиатурное поведение прежнее: `a/y/e/д-эквиваленты` — approve,
   `d/n/…` — deny, Ctrl+C — deny + quit. **Изменение:** голый Enter теперь
   активирует выделенную кнопку (по умолчанию Deny), а не approve — это
   безопаснее; отразить в help-баре.
5. Нижняя панель ввода в режиме Confirming показывает приглушённую строку
   `waiting for confirmation…` — layout не прыгает.

**DoD:**
- Подтверждение — модал по центру, кликабельные кнопки Approve/Deny работают.
- Tab/стрелки переключают кнопку, Enter активирует, буквенные хоткеи работают.
- Деструктивная команда визуально отличается (красная рамка + предупреждение).
- Layout остальных зон при появлении модала не смещается.

## Задача 6. Сворачиваемые блоки команд + клики по ним

**Контекст.** Вывод команды обрезается на 30 строках навсегда. Делаем
expand/collapse: по умолчанию показываем 5 строк, остальное — по клику.

**Шаги:**

1. В `App`: `pub collapsed_overrides: HashMap<usize, bool>` (индекс блока →
   пользователь явно свернул/развернул). Дефолт: блок `Command` с выводом
   длиннее 6 строк — свёрнут до 5 строк + строка-переключатель.
2. В `layout_cache` при построении: для свёрнутого блока после 5 строк вывода
   добавлять строку `▸ … 42 more lines — click to expand` (`fg_muted`, регион
   `OutputToggle`); для развёрнутого длинного — после вывода строку
   `▾ collapse` (`fg_muted`). Прежний жёсткий лимит 30 строк убрать (кэш
   выдерживает; для развёрнутых блоков оставить страховочный потолок 400 строк
   с пометкой `… truncated`).
3. `handle_mouse`: клик по строке региона `OutputToggle` → инвертировать
   состояние в `collapsed_overrides`, инкрементировать `message_rev`
   (перестройка кэша). Клик по строке-заголовку `Command` — тоже toggle
   (удобная большая цель для клика).
4. Заголовок блока команды сделать компактнее и информативнее:
   `▸ $ docker ps -a   ✓` (свёрнут) / `▾ $ docker ps -a   ✓` (развёрнут);
   `✓` — success, `✗` — danger для denied; если терминал conhost — ASCII
   fallback `[ok]/[no]` (использовать существующий подход `strip_emoji`,
   символы `▸▾✓✗` входят в whitelist-диапазоны — проверить, при необходимости
   добавить диапазон 0x2713–0x2717).

**DoD:**
- Длинный вывод по умолчанию свёрнут до 5 строк, клик разворачивает/сворачивает.
- Состояние сохраняется при скролле и новых сообщениях (индексы не съезжают).
- Заголовок команды показывает статус и стрелку-индикатор.

## Задача 7. Стриминг ответа LLM + спиннер

**Контекст.** Это самое заметное «осовременивание»: сейчас пользователь смотрит
на статичное «Agent is thinking...» до полного ответа. GLM API поддерживает
SSE-стриминг (`"stream": true`), reqwest уже собран с feature `stream`.

**Шаги:**

1. `crates/agent/src/glm.rs`: добавить метод стриминга. В трейт `LlmClient`
   добавить метод с дефолтной реализацией-фоллбэком (чтобы не ломать другие
   реализации/моки):
   ```rust
   async fn chat_stream(&self, req: ..., on_delta: &(dyn Fn(&str) + Send + Sync))
       -> Result<LlmResponse> { self.chat(req).await } // дефолт: без стрима
   ```
   В GLM-клиенте: `"stream": true`, читать `bytes_stream()`, парсить SSE-строки
   `data: {...}` построчно (буферизуя разрывы чанков), извлекать
   `choices[0].delta.content` → `on_delta(text)`; накапливать полный текст и
   tool_calls (дельты tool_calls аккумулировать по `index`). Строка `data: [DONE]`
   завершает поток. Вернуть собранный `LlmResponse`, идентичный нестримовому.
2. `crates/tui/src/event.rs`: новое событие `AgentEvent::TextDelta(String)`.
3. `crates/agent/src/agent.rs` / `runner.rs::spawn_agent`: пробросить колбэк,
   который шлёт `TextDelta` в канал. (Если проще архитектурно — передать
   `UnboundedSender<AgentEvent>` в builder агента как опциональный
   `on_text_delta`.)
4. `app.rs::handle_agent_event`:
   - `TextDelta(s)`: если последний блок — «стримящийся» `Agent`, дописать `s`
     к нему; иначе создать новый `ChatBlock::Agent(s)` и флаг
     `pub streaming: bool` в `App`. Инкремент `message_rev` (кэш перестроит
     только хвост — допустимо перестроить целиком, кэш быстрый);
     при `scroll == 0` держать прижатым к низу; при `scroll > 0` — не дёргать
     (пользователь читает историю), просто растёт счётчик `↓ N new`.
   - `Finished`: заменить/финализировать стримящийся блок финальным текстом
     (LLM мог вернуть текст+tool_call — финальный текст источник истины),
     `streaming = false`.
5. Спиннер: в `App` поле `pub tick: u64`, инкрементировать в
   `render_interval.tick()` (тикать даже без `needs_redraw`, когда
   `mode == Thinking` — иначе спиннер замрёт; т.е. условие тика:
   `needs_redraw || app.mode == AppMode::Thinking`). В статус-баре и в панели
   ввода в режиме Thinking: кадры `⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏` (braille); ASCII-fallback
   `|/-\` если braille вне whitelist (добавить 0x2800–0x28FF в whitelist —
   современные Windows Terminal рендерят braille корректно).
   Текст: `⠹ thinking…` → после первой дельты `⠹ writing…`.
6. Панель ввода в Thinking больше не жёлтая коробка «Agent is thinking», а
   обычное поле ввода в disabled-виде (текст `fg_muted`), со спиннером слева —
   layout стабилен.

**DoD:**
- Ответ агента появляется по мере генерации, чат автоскроллится, если был внизу.
- Спиннер анимируется плавно, CPU в простое не растёт (тик только в Thinking).
- Tool calls работают как прежде (стрим корректно собирает tool_calls).
- При ошибке сети посреди стрима — `AgentEvent::Error`, частичный текст остаётся
  в чате с пометкой `System("response interrupted")`.
- Unit-тест SSE-парсера: скормить байтовые чанки с разрывом посреди `data:`-строки,
  проверить корректную сборку дельт и tool_calls.

## Задача 8. Визуальный редизайн чата: карточки вместо рамок, markdown-lite

**Шаги:**

1. **Главный layout:** убрать `Borders::ALL` вокруг чата. Новый вид:
   ```text
    filar ▸ user@host:22          ⠹ thinking      always
   ──────────────────────────────────────────────────────
     you                                                  ← accent, bold
     Проверь свободное место на диске
                                                          ← пустая строка
     agent                                                ← success, bold
     Проверяю диски…
                                                          
     ▾ $ Get-PSDrive -PSProvider FileSystem  ✓            ← warning
     │ Name  Used(GB)  Free(GB)                           ← fg_dim, gutter │

   ──────────────────────────────────────────────────────

   - Статус-бар: без сплошной заливки `DarkGray` во всю ширину. Слева
     `filar ▸ {target}` (accent — только имя таргета), по центру/справа —
     индикатор режима словом и цветом (Normal — ничего не показывать,
     Thinking — спиннер+`warning`, Confirm — `danger`), справа —
     `confirm_mode` тускло. Разделитель под баром — строка `─` цветом
     `fg_muted`.
   - Роль-заголовки: строчные `you` / `agent` без символов-префиксов,
     bold + цвет роли. Тело — обычный `fg` с отступом 1 пробел.
   - Блок команды: заголовок из задачи 6; строки вывода — с gutter `│ `
     цветом `fg_muted` вместо `  | `.
   - `System` — одна строка `· text` (`fg_muted`), `Error` — `✗ text` / `! text`
     (danger).
   - Help-бар: без заливки фоном; клавиши — `fg_dim`, описания — `fg_muted`,
     разделение тремя пробелами. Клик по элементу help-бара выполняет действие
     (hit-test по прямоугольникам слов: `^T` → toggle interactive, `^C` → quit
     и т.д.) — заполнить `app.helpbar_zones: Vec<(Rect, HelpAction)>` при рендере.
2. **Поле ввода:** без рамки. Строка-разделитель сверху и снизу, слева промпт
   `❯ ` (accent; ASCII fallback `> `). Плейсхолдер `введите сообщение…` —
   `fg_muted`. При вводе `!` в начале — промпт меняется на `$ ` и цвет ввода
   на `warning` (мгновенная обратная связь «это shell-команда»). Многострочный
   рост: до 5 строк (`Constraint::Length` вычислять от wrap ввода), дальше —
   внутренний скролл к курсору. PasswordInput — промпт `🔒 `/`* `, маска как
   сейчас.
3. **Markdown-lite для сообщений `Agent`** (только инлайн, без полного парсера):
   - \`code span\` → стиль: `fg` на фоне `surface` (или accent без фона, если
     фон выглядит грязно в conhost);
   - `**bold**` → `Modifier::BOLD`;
   - fenced-блоки ``` ``` ``` → строки с gutter `│ ` и цветом `fg_dim`,
     первая строка — язык тусклым;
   - заголовки `#`/`##` в начале строки → bold accent, символы `#` убрать;
   - маркеры списков `- ` → `• `.
   Реализовать в `ui/text.rs` как `fn render_markdown_line(&str, theme) -> Vec<Span>`
   + распознавание fence-состояния на уровне блока. Не поддерживаем: таблицы,
   ссылки, вложенность — сознательно, ради лёгкости.
4. Проверить всё в Windows Terminal и conhost; для conhost предусмотреть
   ASCII-фоллбэки (`❯`→`>`, `·`→`-`, `│`→`|`, `✓`→`[ok]`). Определение: env
   `WT_SESSION` есть → современный терминал; иначе — консервативный набор.
   Вынести в `theme.rs` структуру `Glyphs { prompt: &str, gutter: &str, ... }`.

**DoD:**
- Ни одной сплошной рамки в основном экране (кроме модала подтверждения).
- Скриншот-проверка: чат читается «по ролям» без напряжения, цвета не кричат.
- Markdown-lite: код-спаны, bold, fence-блоки, заголовки, списки рендерятся.
- В conhost всё отображается без «?» (ASCII-фоллбэки работают).
- Unit-тесты `render_markdown_line`: code span, bold, смешанный случай,
  незакрытый маркер (рендерится как обычный текст).

## Задача 9. Выделение текста мышью и копирование в буфер

**Контекст.** Главная боль mouse-capture в TUI: пропадает нативное выделение
терминала. Возвращаем его своими руками — иначе пользователь не сможет
скопировать вывод команды.

**Шаги:**

1. Добавить зависимость `arboard = "3"` в `crates/tui/Cargo.toml`
   (кроссплатформенный clipboard; Windows поддерживается из коробки).
2. В `App`: `pub selection: Option<Selection>`, где
   `struct Selection { anchor: (usize /*line*/, usize /*col*/), head: (usize, usize) }` —
   координаты в пространстве строк `layout_cache` (не экрана!), чтобы выделение
   переживало скролл.
3. `handle_mouse`:
   - `Down(Left)` в области чата (не по toggle/скроллбару) → `selection = Some`
     с anchor=head в позиции клика, `mouse_drag = Some(Selection)`;
   - `Drag(Left)` → обновлять `head`; если курсор у верхней/нижней кромки
     области чата — автоскролл на 1 строку за тик;
   - `Up(Left)`: если anchor == head → снять выделение (обычный клик).
4. Рендер: в `chat.rs` после получения видимых строк накладывать
   `theme.selection_bg` на диапазон выделения (пересечение стилей span-ов:
   проще всего пере-стилизовать посимвольно только затронутые строки).
5. Копирование: `Ctrl+Shift+C` или автоматически по `Up` (выбрать: авто-копия
   по отпусканию — поведение как в терминалах с «copy on select»; сделать
   авто-копию, это элегантнее). Извлечь текст выделенных строк из
   `layout_cache` (сырое содержимое `Line` → `to_string`), для средних строк —
   целиком, для крайних — по колонкам. После копирования — короткий тост:
   строка `· copied` в статус-баре на ~1.5 сек (хранить `toast: Option<(String, Instant)>`).
6. Двойной клик — выделить слово, тройной — строку (отслеживать интервал
   < 400 мс между Down по той же позиции).
7. Любое новое сообщение/перестройка кэша со смещением индексов → снять
   выделение (просто и предсказуемо).

**DoD:**
- Drag мышью выделяет текст в чате, отпускание копирует в системный буфер,
  появляется тост `copied`.
- Двойной клик выделяет слово, тройной — строку.
- Выделение переживает скролл колесом; сбрасывается при новых сообщениях.
- Сборка на Windows проходит, clipboard работает (ручная проверка).

## Задача 10. Мышь в интерактивном режиме терминала

**Контекст.** В режиме Interactive (Ctrl+T) свой эмулятор на `alacritty_terminal`.
Нужно: скролл истории терминала колесом и проброс мыши в приложения на удалёнке
(vim, htop, mc), если они её запросили.

**Шаги:**

1. В `TerminalModel` (terminal.rs): API для scrollback —
   `pub fn scroll_display(&mut self, delta: i32)` через
   `term.scroll_display(Scroll::Delta(delta))`, и `pub fn mouse_mode(&self) -> bool`
   по `term.mode()` (`TermMode::MOUSE_MODE`-флаги: REPORT_CLICK/DRAG/MOTION + SGR).
2. `handle_mouse` в режиме Interactive:
   - если приложение в терминале запросило мышь (`mouse_mode() == true`) —
     кодировать событие в SGR-последовательность (`\x1b[<b;x;yM/m`, координаты
     1-based относительно области терминала) и слать в `pending_term_input`
     (в терминах существующего механизма проброса байтов);
   - иначе: колесо → `scroll_display(±3)` (просмотр истории), любой ввод с
     клавиатуры → сброс скролла в низ (`Scroll::Bottom`).
3. Учесть alternate screen: в alt-screen колесо без mouse-mode транслировать
   в стрелки ↑↑↑/↓↓↓ (по 3 на тик) — стандартное поведение эмуляторов, чтобы
   колесо «просто работало» в less/man.
4. В help-баре режима Interactive добавить `wheel scroll`.

**DoD:**
- Колесо скроллит историю локального PowerShell в интерактивном режиме.
- В `htop`/`mc` по SSH (тест через docker/sshd из репо) клики и колесо
  доходят до приложения.
- В `less` колесо листает (трансляция в стрелки).

## Чего сознательно НЕ делаем (anti-scope)

- Никаких вкладок, панелей файлового менеджера, деревьев, статус-виджетов CPU/RAM.
- Никаких тем на выбор в UI (одна выверенная тёмная тема; светлую — потом).
- Никакого полного markdown (таблицы, картинки, ссылки).
- Никаких эффектов: градиенты, тени, «печатающаяся машинка» с задержкой,
  анимации появления блоков.
- Command palette, автодополнение команд — вне рамок этого плана (кандидаты
  на следующую итерацию).
