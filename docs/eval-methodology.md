# Тестирование ИИ-агентов и моделей: методика eval для команд

*Практический туториал: как перестать спорить «какая модель лучше» и начать измерять. Пошагово — от подготовки датасета до регрессионных прогонов, с конфигами и примерами.*

---

## 1. Зачем это нужно

Типичная ситуация: подразделение годами оттачивало промпты, агентов и спеки под конкретную модель. Приходит новая модель — и «она хуже». Хуже ли модель на самом деле, или промпты переобучены под старую? Без измерений этот спор не решается: каждая сторона опирается на впечатления от нескольких запусков.

Eval (evaluation) — это способ превратить впечатления в цифры. Идея простая:

1. Команда собирает **golden dataset** — набор реальных задач со своими критериями приемки.
2. Один и тот же датасет прогоняется через несколько моделей **одним и тем же харнессом**.
3. Ответы оцениваются **одинаковыми грейдерами** (автопроверки, модель-судья, выборочно人 человек).
4. Решение принимается по метрикам: качество, латентность, стоимость, доля отказов.

Побочный эффект важнее основного: датасет остаётся жить. Любая будущая смена модели, промпта или обвязки прогоняется через тот же eval как регрессионный тест. Спор «стало хуже» превращается в diff двух отчётов.

**Ограничение честности.** Eval не даёт абсолютной истины — он даёт воспроизводимое сравнение на *ваших* задачах. Модель, выигравшая на датасете команды A, может проиграть на датасете команды B. Это нормально и правильно: выбирать модель нужно под свою нагрузку, а не по чужим бенчмаркам.

## 2. Архитектура: почему шлюз решает половину проблемы

Если доступ к моделям идёт через LiteLLM-шлюз с алиасами, у вас уже есть идеальный полигон для A/B-сравнений:

```
датасет ──► харнесс (promptfoo) ──► шлюз LiteLLM ──► glm-5.2
                                            ├───────► glm-5.1
                                            ├───────► glm-flashx
                                            └───────► (другие алиасы)
```

Харнесс не знает и не должен знать, где физически крутится модель — он просто дёргает разные алиасы одного OpenAI-совместимого эндпоинта. Это даёт три бонуса:

- **Идентичные условия.** Все модели получают ровно один и тот же запрос через один и тот же транспорт.
- **Учёт стоимости бесплатно.** Шлюз уже считает спенд по каждому запросу — стоимость задачи берётся из его логов, а не оценивается на глаз.
- **Никакой возни с ключами.** Один виртуальный ключ на eval-проект с отдельным бюджетом — и расход экспериментов не смешивается с рабочим.

> **Совет:** заведите под eval отдельный виртуальный ключ с собственным бюджетом (например, $10–20 на цикл тестирования) и алиасом вроде `key_alias: eval-runner`. Так эксперименты видны в логах отдельной строкой.

## 3. Роли и границы ответственности

Чтобы методика прижилась, зафиксируйте, кто чем владеет:

| Роль | Владеет | Не лезет в |
|---|---|---|
| Платформенная команда | Харнесс, шлюз, методика, общий дашборд, модель-судья | Содержание датасетов команд |
| Подразделение | Свой датасет, свои рубрики, интерпретация результатов по своим задачам | Настройки харнесса и шлюза |
| Арбитраж (техлид/архитектор) | Правила честного сравнения, разрешение споров по рубрикам | Повседневные прогоны |

Датасет — собственность команды. Никто снаружи не знает лучше неё, что такое «хороший ответ» в её домене. Задача платформы — дать инструмент и правила игры, а не оценки.

## 4. Шаг 1. Готовим golden dataset

Это самый трудоёмкий и самый ценный этап. Всё остальное — механика.

### 4.1. Откуда брать задачи

Только из реальной работы. Источники:

- Логи реальных запросов к агенту за последние недели (шлюз их хранит).
- Тикеты/жалобы: «модель не справилась с X» — каждый такой случай это готовый кейс.
- Типовые операции флоу: если агент каждый день сравнивает ответы, классифицирует заявки, генерирует тесты — берите по 5–10 представителей каждой операции.

Синтетические задачи («придумаем что-нибудь похожее») — в последнюю очередь и только для покрытия редких веток. Синтетика систематически проще реальности.

### 4.2. Сколько нужно

- **Минимум для старта: 20 задач.** Меньше — шум перевесит сигнал.
- **Рабочий объём: 30–50.** Достаточно для уверенных выводов по одному флоу.
- Если у команды несколько разных флоу — по 20+ на каждый, и оценивать их раздельно. Усреднение по разнородным задачам прячет проблемы.

### 4.3. Структура кейса

Каждый кейс — это три вещи:

1. **Вход**: промпт + контекст, ровно в том виде, в каком его получает модель в бою (включая системный промпт агента).
2. **Критерий приемки** — один из трёх типов:
   - *Эталон*: точный ожидаемый ответ (для задач с однозначным результатом — извлечение, классификация, расчёт).
   - *Чеклист*: список того, что обязано присутствовать/отсутствовать в ответе.
   - *Рубрика*: текстовое описание, что такое хороший ответ, для оценки моделью-судьёй (для открытых задач — суммаризация, сравнение, генерация).
3. **Метаданные**: тип задачи, источник, сложность (хотя бы subjective easy/medium/hard).

### 4.4. Формат хранения

Практичный вариант — CSV или YAML в git-репозитории рядом с конфигом харнесса. Пример CSV для простых задач (промпт + ожидание):

```csv
task_id,category,question,__expected
cmp-001,compare,"Сравни два ответа ниже и укажи, какой точнее отвечает на вопрос пользователя... [текст]","contains: Ответ Б"
ext-014,extract,"Извлеки из текста заявки номер договора и дату... [текст]","regex: ДГ-\d{6}"
cls-007,classify,"Определи категорию обращения... [текст]","equals: технический сбой"
```

Для сложных кейсов с длинным контекстом и рубриками удобнее YAML (см. раздел 7).

### 4.5. Чего не делать

- **Не подгонять кейсы под известные ответы моделей.** Датасет собирается до прогонов, вслепую.
- **Не смешивать в одном датасете разные флоу.** «Средняя температура» по генерации кода и классификации тикетов не значит ничего.
- **Не включать задачи без критерия приемки.** «Посмотрим, что ответит» — это не тест.
- **Не хранить в датасете секреты и персональные данные.** Обезличьте реальные кейсы перед коммитом.

## 5. Шаг 2. Устанавливаем харнесс

Используем **promptfoo** — открытый инструмент для матричных прогонов «промпты × модели × тесты». Бесплатный, локальный, работает с любым OpenAI-совместимым эндпоинтом, то есть с нашим шлюзом напрямую.

### 5.1. Установка

Нужен Node.js 18+.

```bash
# вариант 1: глобально
npm install -g promptfoo

# вариант 2: без установки, через npx
npx promptfoo@latest --version
```

### 5.2. Инициализация проекта

```bash
mkdir team-eval && cd team-eval
npx promptfoo@latest init
```

Появится заготовка `promptfooconfig.yaml`. Дальше заменим её на свою.

### 5.3. Структура проекта

Рекомендуемая раскладка (кладётся в git):

```
team-eval/
├── promptfooconfig.yaml      # главный конфиг: модели, тесты, судья
├── prompts/
│   ├── agent-system.txt      # системный промпт агента (как в бою)
│   └── task.txt              # шаблон пользовательского запроса
├── datasets/
│   ├── compare.csv           # кейсы: сравнение ответов
│   ├── extract.csv           # кейсы: извлечение данных
│   └── longctx.yaml          # кейсы с длинным контекстом и рубриками
└── README.md                 # кто владелец, как гонять, как читать отчёт
```

## 6. Шаг 3. Конфигурируем модели (провайдеры)

Все модели подключаются через шлюз. Ключ кладём в переменную окружения, в конфиге его нет.

```bash
export EVAL_GATEWAY_KEY="sk-ваш-виртуальный-ключ-для-eval"
export EVAL_GATEWAY_URL="https://ваш-шлюз:4000/v1"
```

Блок провайдеров в `promptfooconfig.yaml` — на примере алиасов семейства GLM (добавьте свои по аналогии):

```yaml
providers:
  - id: openai:chat:glm-5.2
    label: GLM-5.2 (1M ctx)
    config:
      apiBaseUrl: ${EVAL_GATEWAY_URL}
      apiKey: ${EVAL_GATEWAY_KEY}
      temperature: 0
      max_tokens: 4096
      # таймаут щедрый: модель в режиме глубокого рассуждения думает долго,
      # и резать её на 60с — значит воспроизвести ложные "таймауты"
      timeoutMs: 600000

  - id: openai:chat:glm-5.1
    label: GLM-5.1
    config:
      apiBaseUrl: ${EVAL_GATEWAY_URL}
      apiKey: ${EVAL_GATEWAY_KEY}
      temperature: 0
      max_tokens: 4096
      timeoutMs: 600000

  - id: openai:chat:glm-flashx
    label: GLM FlashX (дешёвая)
    config:
      apiBaseUrl: ${EVAL_GATEWAY_URL}
      apiKey: ${EVAL_GATEWAY_KEY}
      temperature: 0
      max_tokens: 4096
      timeoutMs: 300000

  # сюда добавите остальные алиасы вашего шлюза:
  # corp-qwen, ds-pro и т.д. — тот же шаблон, меняется только id
```

Пояснения к неочевидному:

- **`temperature: 0`** — для сравнений нужна воспроизводимость. Творческие задачи можно гонять и на рабочей температуре, но тогда каждую задачу прогоняйте 3–5 раз и смотрите разброс.
- **`timeoutMs`** — выставляйте с запасом под «думающие» модели. Значительная часть жалоб «модель отваливается» в реальности — это таймаут клиента, а не сбой модели. В eval мы хотим измерить реальную латентность, а не воспроизвести обрезание.
- **`label`** — то, что будет в отчёте; давайте говорящие имена.

## 7. Шаг 4. Описываем промпты и тесты

### 7.1. Промпт — как в бою

Ключевое правило: тестировать надо тот промпт, который реально работает в агенте, включая системную часть. `prompts/agent-system.txt` — копия боевого системного промпта. Шаблон запроса с переменными:

```
# prompts/task.txt
{{question}}
```

В конфиге:

```yaml
prompts:
  - file://prompts/task.txt

defaultTest:
  options:
    # системный промпт агента подставляется каждому тесту
    # (можно и через chat-формат промпта, см. доку promptfoo)
    ...
```

Либо, что часто удобнее, промпт описывается в chat-формате одним файлом JSON/YAML с ролями system/user — тогда системная часть живёт прямо в нём.

### 7.2. Тесты из CSV

Простые задачи с детерминированной проверкой подключаются одной строкой:

```yaml
tests: file://datasets/compare.csv
```

Колонка `__expected` в CSV понимает встроенные проверки: `equals:`, `contains:`, `icontains:`, `regex:`, `is-json` и другие. Это самый дешёвый тип грейдера — используйте его везде, где ответ проверяется формально.

### 7.3. Тесты с рубрикой (LLM-as-judge)

Для открытых задач критерий описывается рубрикой, а оценивает модель-судья:

```yaml
tests:
  - description: "Сравнение двух ответов на пользовательский вопрос"
    vars:
      question: file://datasets/cases/cmp-002-input.txt
    assert:
      - type: llm-rubric
        value: |
          Оцени ответ по рубрике (каждый пункт: да/нет):
          1. Явно назван более точный из двух ответов.
          2. Выбор обоснован ссылкой на конкретные фрагменты.
          3. Нет фактических ошибок в пересказе сравниваемых ответов.
          4. Вывод однозначный, без "оба хороши по-своему", если ответы
             объективно различаются по точности.
          Вердикт PASS, если выполнены пункты 1, 3 и 4.
```

Судья настраивается глобально — и вот **критичное правило: судьёй берём модель, не участвующую в сравнении.** Модели систематически завышают оценки текстам «своего» семейства; судья-участник — это конфликт интересов.

```yaml
defaultTest:
  options:
    provider:
      # судья — отдельный алиас на шлюзе, НЕ из сравниваемых
      id: openai:chat:judge-model
      config:
        apiBaseUrl: ${EVAL_GATEWAY_URL}
        apiKey: ${EVAL_GATEWAY_KEY}
        temperature: 0
```

### 7.4. Кейсы с длинным контекстом

Для задач, где вход — десятки тысяч токенов (агент по репозиторию, RLM-подобные обвязки), кейсы храните в YAML со ссылками на файлы контекста:

```yaml
# datasets/longctx.yaml
- description: "Анализ большого документа: свод противоречий"
  vars:
    question: file://cases/longctx-001-prompt.txt
  assert:
    - type: llm-rubric
      value: file://cases/longctx-001-rubric.txt
    - type: latency
      threshold: 300000   # p95-ожидание: ответ быстрее 5 минут
```

Здесь же уместно проверять именно то, ради чего берётся модель с большим окном: положите в контекст «иголку» (конкретный факт в середине документа) и спросите про неё — это тест на реальную работу длинного контекста, а не на заявленный размер окна.

## 8. Шаг 5. Запуск и чтение результатов

### 8.1. Прогон

```bash
npx promptfoo@latest eval
```

Матрица «каждый тест × каждая модель» выполняется параллельно. По завершении:

```bash
npx promptfoo@latest view
```

— открывается локальный веб-отчёт: таблица моделей по тестам, PASS/FAIL по каждой ячейке, латентность, ответы рядом для глазного сравнения.

Экспорт для конфы/отчёта:

```bash
npx promptfoo@latest eval -o results.html   # или .csv / .json
```

### 8.2. Какие метрики смотреть

Решение принимается по четырём осям, не по одной:

| Метрика | Откуда | На что отвечает |
|---|---|---|
| Pass rate (доля успешных) | Отчёт promptfoo | Качество на ваших задачах |
| Латентность p50 / p95 | Отчёт promptfoo | Уложится ли в таймауты флоу; «отваливания» видны здесь |
| Стоимость на задачу | Логи шлюза (spend по eval-ключу / число задач) | Экономика при масштабировании |
| Доля технических отказов | Отчёт (ошибки, таймауты) | Стабильность интеграции |

Модель, которая на 5% хуже по pass rate, но втрое дешевле и вдвое быстрее — это не «хуже», это другой профиль. Выбор становится осознанным компромиссом, зафиксированным цифрами.

### 8.3. Калибровка судьи

LLM-судья — тоже модель, и ему нельзя верить слепо:

1. Возьмите 10–15% провалов и столько же прохождений.
2. Прогоните их через человека (владельца датасета) вслепую — без вердикта судьи.
3. Посчитайте согласие человек/судья. Если ниже ~85% — рубрики слишком расплывчаты, переписывайте их конкретнее (лучший приём: заменять «ответ полезен» на проверяемые пункты «назван X», «нет утверждения Y»).

## 9. Правило честного сравнения (самое важное в методике)

Проблема, ради которой всё затевалось: промпты команд переобучены под старую модель. Сравнивать новую модель на промпте, годами вылизанном под старую, — нечестно по отношению к новой. Бесконечно тюнить промпт под новую — нечестно по отношению к старой. Компромисс фиксируется правилом:

> **Каждой модели разрешается ровно один раунд адаптации промпта силами команды.**
>
> Прогон делается дважды: (1) baseline — боевой промпт как есть, на всех моделях; (2) adapted — каждая модель на промпте, адаптированном под неё за один заход (формат, системная инструкция, структура — без изменения сути задачи).

Оба результата попадают в отчёт. Разница между baseline и adapted — отдельная ценная метрика: она показывает, насколько команда вообще привязана к формату конкретной модели. Большой разрыв — сигнал, что промпты стоит переписать более переносимо.

В promptfoo это делается матрицей промптов:

```yaml
prompts:
  - id: baseline
    file://prompts/task-baseline.txt
  - id: adapted-glm
    file://prompts/task-adapted-glm.txt
```

## 10. Регрессионный режим: eval как CI

Первичное сравнение — это только начало. Главная ценность — регрессия:

- Датасет и конфиг живут в git рядом с агентом.
- Любое изменение (модель, промпт, версия обвязки, параметры шлюза) → прогон eval → diff отчётов.
- Порог включается прямо в конфиг: прогон падает, если pass rate ниже заданного.

```yaml
# опционально: жёсткий порог для CI
evaluateOptions:
  maxConcurrency: 4
tests:
  # ... ваши тесты
# в CI: promptfoo eval --fail-on-error, анализ results.json скриптом
```

Практика: гонять полный датасет на каждый коммит дорого — заведите **smoke-набор** из 10 самых показательных кейсов для частых прогонов и полный набор для релизных решений.

И отдельное правило гигиены: **датасет пополняется из инцидентов.** Каждый случай «модель ответила плохо» из боевой эксплуатации оформляется как новый кейс с критерием приемки. Через полгода у команды будет датасет, точно описывающий её реальные болевые точки.

## 11. Типовые ловушки

**«Модель отваливается по таймауту».** Проверьте три места, прежде чем винить модель: таймаут клиента/обвязки, `request_timeout` шлюза, `timeoutMs` харнесса. Думающие режимы флагманских моделей легитимно работают минуты. В eval-отчёте честная латентность видна как есть — и если p95 задачи 4 минуты, то это факт для инженерного решения (поднять таймаут / выбрать быстрый режим / взять модель полегче), а не «нестабильность».

**«Лимит кончился — модель плохая».** Исчерпание бюджета — это факт об интенсивности использования и настройке лимитов, а не о качестве модели. В отчёт eval бюджеты не входят вообще; их обсуждают отдельно, по логам спенда.

**Сравнение разных классов моделей в лоб.** Компактная корпоративная модель на своей узкой задаче с заточенным промптом может обыгрывать флагман — это нормальный результат, а не парадокс. Флагманы берут широтой и сложными задачами. Поэтому в отчёте всегда смотрите разрезы по категориям задач, а не только общий счёт.

**Судья из семейства участника.** Завышение «своим» — документированный эффект. Судья всегда нейтральный.

**Один прогон при temperature > 0.** Разброс между запусками может быть больше разницы между моделями. Либо temperature 0, либо серия прогонов с усреднением.

**Датасет «под ответ».** Если кейсы отбирались после того, как посмотрели, где какая модель ошибается, — сравнение испорчено. Датасет собирается вслепую и замораживается до прогонов.

## 12. План пилота (2–3 недели)

**Неделя 1 — датасет.** Выбрать одно подразделение с живым конфликтом «старая vs новая модель». Владелец флоу собирает 25–40 кейсов из реальной работы по форматам из раздела 4. Платформа параллельно поднимает проект харнесса и eval-ключ на шлюзе.

**Неделя 2 — прогоны.** Baseline-прогон на всех моделях → один раунд адаптации промптов → adapted-прогон. Калибровка судьи на выборке. Сведение метрик: качество / латентность / стоимость / отказы.

**Неделя 3 — разбор и стандарт.** Встреча с командой: разбор отчёта, решение по модели (или по гибриду «этот класс задач — на эту модель»). Фиксация датасета как регрессионного. Оформление методики как внутреннего стандарта и тираж на следующие подразделения.

Критерий успеха пилота — не «победила модель X», а: команда согласна с процедурой измерения, датасет живёт в git, следующий спор о моделях начинается с фразы «прогоните через eval».
