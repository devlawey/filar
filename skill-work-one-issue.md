# Skill: work-one-issue

Обработать одну issue из milestone по флоу «одна issue = одна ветка = один PR» и
остановиться на ревью. Универсально для любой issue: всё специфичное берётся из тела
самой issue; инварианты проекта — из AGENTS.md.

**Параметр:** `ISSUE_NUMBER` — номер issue (если не передан, см. шаг 1).

---

## Промпт

```
Ты работаешь над репозиторием filar по флоу «одна issue = одна ветка = один PR».
Соблюдай правила из AGENTS.md в корне репозитория. Работай строго над ОДНОЙ issue
за раз и останавливайся на ревью.

ВХОД: номер issue — {{ISSUE_NUMBER}}.
(Если номер не передан — возьми из целевого milestone самую приоритетную ещё
открытую issue, не находящуюся в работе, и сообщи мне, какую выбрал.)

ШАГИ:

1. Прочитай тело issue #{{ISSUE_NUMBER}} полностью. Оно — единственный источник
   правды: что менять, в каких файлах и каковы критерии готовности (Definition of
   Done). Не придумывай требований сверх описанных; если в issue чего-то критично
   не хватает или оно противоречиво — не угадывай, остановись и спроси меня.

2. Создай ветку от актуального `main`. Имя: `fix/{{ISSUE_NUMBER}}-<краткий-слаг>`,
   где слаг — 2-4 слова из заголовка issue в kebab-case (например `fix/3-cancel-task`).
   Префикс выбери по характеру: `fix/`, `feat/` или `refactor/`.

3. Реализуй изменения строго по телу issue. Соблюдай инварианты проекта из AGENTS.md
   (zero-install на удалёнке, никаких команд без подтверждения, секреты только из env,
   изоляция за трейтами). Меняй только то, что относится к этой issue — не примешивай
   посторонние правки.

4. Если в критериях готовности issue есть тесты — добавь/обнови их. Прогони
   `cargo build --workspace` и `cargo test --workspace`: сборка и юнит-тесты должны
   быть зелёными. Тесты, помеченные `#[ignore]` (требуют docker-sshd), сам не запускай
   и не поднимай инфраструктуру — перечисли их в описании PR как «требуют ручной проверки».

5. Обнови `PROGRESS.md` в корне репозитория: кратко отметь, что сделано по этой
   issue, любые принятые решения и изменения публичных контрактов (сигнатуры
   трейтов и т.п.). Это входит в ТУ ЖЕ правку (тот же коммит/PR), не отдельным проходом.

6. Сделай ОДИН логический коммит (включающий и правку кода, и обновление
   PROGRESS.md). Сообщение в Conventional Commits:
   `<type>(<scope>): <краткое описание>` (например `fix(ssh): ...`,
   `refactor(agent): ...`). Тип и суть бери из характера правки.

7. Запушь ветку и открой Pull Request в `main`. В описании PR обязательно:
   - строка `Closes #{{ISSUE_NUMBER}}` (чтобы issue закрылась при мерже);
   - краткое summary что и зачем изменено;
   - явная пометка про тесты: что прошло автоматически и что требует ручной
     проверки (docker-sshd), если такие есть;
   - список того, что НЕ вошло в скоуп, если по ходу всплыло смежное.

8. ОСТАНОВИСЬ. Не мержи PR. Не бери следующую issue. Жди моего ревью и ревью
   CodeRabbit. К следующей задаче переходи только после того, как я смержу этот PR.

ОГРАНИЧЕНИЯ:
- Не трогай ветку `main` напрямую и не делай force-push.
- Не закрывай и не редактируй другие issue.
- Один PR решает ровно одну issue.
```

---

## Как пользоваться

1. Убедись, что `main` актуален, а `AGENTS.md` и `.coderabbit.yaml` закоммичены.
2. Запусти скилл, передав `ISSUE_NUMBER` (или без него — агент выберет приоритетную).
3. Держи агента на подтверждении вызовов GitHub MCP, пока не привыкнешь к флоу.
4. После открытия PR дождись ревью CodeRabbit, при необходимости попроси агента
   поправить по комментариям, затем смержи PR сам. Issue закроется автоматически.
5. К следующей issue переходи только после мержа предыдущего PR.
