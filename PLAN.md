# План разработки: терминал с агентом поверх SSH (Rust)

> Документ-задание для агента в IDE. Каждый этап содержит цель, конкретные задачи,
> используемые крейты и **Definition of Done (DoD)** — критерии, по которым этап
> считается завершённым. Этапы упорядочены так, чтобы **главная фича (агент поверх
> SSH с zero-install на удалёнке) была готова и проверяема первой**, а «красивый
> терминал в стиле Warp» наслаивался поверх уже работающего ядра.

---

## 0. Принципы и нефункциональные требования

- **Язык:** Rust (edition 2021), async-рантайм — `tokio`.
- **Главная фича:** агент управляет удалённой машиной через SSH, на удалёнке
  **ничего не устанавливается на диск**. Допускается только эфемерная инъекция
  shell-команд в рантайме сессии.
- **LLM:** GLM 5.1 через HTTP API с ключом из окружения. Интеграция изолирована за
  трейтом `LlmClient`, чтобы модель можно было заменить.
- **Безопасность по умолчанию:** агент не выполняет команды на удалённой машине без
  подтверждения пользователя (режим confirm). Деструктивные команды требуют
  явного approve.
- **Тестируемость:** SSH-слой тестируется против локального `sshd` в Docker, а не
  против продакшн-серверов.

### Структура воркспейса (cargo workspace)

```
warp-lite/
├── Cargo.toml                # workspace
├── crates/
│   ├── core/                 # общие типы, ошибки, конфиг
│   ├── transport/            # абстракция выполнения команд: local PTY + SSH
│   ├── agent/                # LLM-клиент, цикл агента, инструменты (tools)
│   ├── tui/                  # интерфейс (ratatui на старте)
│   └── app/                  # бинарь: связывает всё вместе
└── docker/
    └── sshd/                 # тестовый ssh-сервер
```

Создавать крейты по мере необходимости, но границы держать с самого начала —
особенно `transport` ↔ `agent` ↔ `tui`.

## Этап 1 — Фундамент проекта

**Цель:** скелет воркспейса, конфиг, логирование, обработка ошибок.

**Задачи:**
1. Завести cargo workspace со структурой выше (пока пустые крейты-заглушки).
2. В `core`: типы ошибок (`thiserror`), общий `Result`, загрузка конфига
   (`serde` + TOML), чтение секретов из env (`GLM_API_KEY` и т.п.).
3. Логирование: `tracing` + `tracing-subscriber`, уровень из env.
4. Файл конфига `config.toml`: SSH-таргеты, модель LLM, базовый URL API,
   таймауты, режим подтверждения команд.

**Крейты:** `tokio`, `serde`, `toml`, `thiserror`, `anyhow`, `tracing`,
`tracing-subscriber`.

**DoD:** `cargo build` собирает воркспейс; бинарь `app` стартует, читает конфиг,
пишет структурированный лог и завершается. Секреты не хардкодятся.

## Этап 2 — SSH-ядро (сердце killer-фичи)

**Цель:** установить SSH-соединение и держать **один персистентный shell-канал**,
через который можно выполнять команды с чёткими границами и exit-кодами, сохраняя
состояние (cwd, env) между командами. Zero-install на удалёнке.

**Задачи:**
1. Подключение через `russh`: хост, порт, пользователь.
2. Аутентификация: по ключу из `~/.ssh`, через ssh-agent, и (опц.) по паролю.
   Обработка `known_hosts` / verify host key.
3. Открыть **один** интерактивный shell-канал (request pty + shell) и держать его
   живым весь сеанс.
4. **Протокол маркеров** для выделения границ команд (см. ниже).
5. Асинхронное чтение потока канала, буферизация, парсинг до маркера, извлечение
   `exit_code`.
6. API уровня: `async fn run(&self, cmd: &str) -> CommandResult { stdout, stderr,
   exit_code, duration }`.

### Протокол маркеров (ключевой механизм)

Каждую команду агента оборачиваем так, чтобы получить уникальную метку конца и код
возврата. Базовый вариант для POSIX-шелла:

```sh
<команда>
__printf '\n__WARPLITE__%s__%d__\n' "$REQ_ID" "$?"
```

- `REQ_ID` — уникальный id запроса (UUID/счётчик), исключает коллизии вывода.
- Всё, что пришло **до** строки-маркера, — это вывод команды.
- Из самого маркера парсим `exit_code`.
- Поскольку канал один и живёт постоянно, `cd`, экспортированные переменные и
  состояние шелла **сохраняются** между командами.

> Тонкости, которые нужно учесть в реализации:
> - stdout и stderr в одном канале смешаны — на этом этапе допустимо склеить их;
>   разделение вынести в опциональную задачу (отдельный exec-канал per command
>   через мультиплексирование, если нужна чистая изоляция stderr/exit).
> - Команда может сама печатать строку, похожую на маркер → `REQ_ID` + маловероятный
>   префикс снижают риск; для надёжности можно добавить nonce.
> - Долго висящие/интерактивные команды (запросившие ввод) — предусмотреть таймаут
>   и возможность отправить Ctrl-C в канал.

**Крейты:** `russh`, `russh-keys`, `tokio`, `uuid`.

**DoD:**
- Подключение к тестовому `sshd` (Docker) проходит, host key проверяется.
- `run("pwd")` → корректный cwd; `run("cd /tmp")` затем `run("pwd")` → `/tmp`
  (доказательство сохранения состояния).
- `run("false")` → `exit_code == 1`; `run("echo hi")` → stdout `hi`, `exit_code 0`.
- На удалённой машине **не создано ни одного файла** (проверяется тестом).

## Этап 3 — Абстракция транспорта (local + SSH за одним трейтом)

**Цель:** агент не должен знать, выполняется команда локально или по SSH.

**Задачи:**
1. Трейт `CommandExecutor` с `async fn run(&self, cmd) -> CommandResult` и
   `async fn run_streaming(...)` (стрим вывода чанками для длинных команд).
2. Реализация `SshExecutor` (поверх Этапа 2).
3. Реализация `LocalExecutor` через `portable-pty` (локальный PTY).
4. Единый тип `CommandResult` и потоковые события (`Stdout(chunk)`, `Stderr(chunk)`,
   `Exit(code)`).
5. Возможность прервать выполняющуюся команду (cancel-токен → Ctrl-C/SIGINT).

**Крейты:** `portable-pty`, `tokio`, `bytes`, `futures`.

**DoD:** один и тот же набор интеграционных тестов проходит и на `LocalExecutor`, и
на `SshExecutor`. Стриминг вывода работает для команды, печатающей данные секундами
(напр. `ping`/цикл с `sleep`), и корректно прерывается по cancel.

## Этап 4 — LLM-клиент (GLM 5.1)

**Цель:** изолированный клиент к GLM с поддержкой tool calling.

**Задачи:**
1. Трейт `LlmClient`: `async fn chat(&self, messages, tools) -> LlmResponse`, где
   ответ — это либо текст, либо запрос на вызов инструмента (tool call).
2. Реализация `GlmClient` через `reqwest`: base URL и модель из конфига, ключ из
   env. **Уточнить у поставщика точный endpoint и формат** — GLM-платформа обычно
   даёт OpenAI-совместимый `chat/completions` с function/tool calling; если так —
   переиспользовать OpenAI-совместимую схему запроса/ответа.
3. Сериализация истории сообщений (`system`/`user`/`assistant`/`tool`) и описаний
   инструментов (JSON Schema).
4. Поддержка стриминга ответа (SSE) — опционально, можно начать с non-streaming.
5. Ретраи с backoff, таймауты, аккуратная обработка ошибок API и rate limit.

**Крейты:** `reqwest` (json, stream), `serde`, `serde_json`, `tokio`,
`eventsource-stream` (если SSE).

**DoD:** интеграционный smoke-тест (за фиче-флагом, требует ключ): отправить простой
prompt → получить текстовый ответ; отправить prompt с одним объявленным
инструментом → модель возвращает корректно структурированный tool call.

## Этап 5 — Ядро агента (петля + инструменты + безопасность)

**Цель:** рабочий agent loop, который через LLM выбирает действия, выполняет их
через `CommandExecutor` и возвращает результаты обратно в модель — **с
подтверждением перед выполнением**.

**Задачи:**
1. Определить инструменты (tools), доступные модели, минимум:
   - `run_command { command, explanation }` — выполнить shell-команду на текущем
     таргете.
   - `read_file { path }` / `list_dir { path }` — реализовать как обёртки над
     командами (`cat`, `ls`), чтобы остаться zero-install.
2. **Agent loop:** prompt пользователя → LLM → (tool call?) → подтверждение → выполнение
   через executor → результат как `tool` message → снова LLM → … → финальный
   текстовый ответ.
3. **Слой безопасности:**
   - режим `confirm` (по умолчанию): каждый `run_command` показывается пользователю
     с объяснением и ждёт approve/deny;
   - опц. allowlist «безопасных» read-only команд для авто-подтверждения;
   - явное предупреждение/блок для деструктивных паттернов (`rm -rf`, `dd`,
     перенаправления в системные пути и т.п.).
4. **Управление контекстом:** обрезка/суммаризация длинного вывода команд перед
   отправкой в LLM (вывод на сотни КБ нельзя слать целиком); лимит итераций цикла
   (anti-runaway), чтобы агент не зациклился.
5. Логирование каждого шага (что предложил, что подтвердили, что выполнилось).

**Крейты:** те, что выше; плюс `serde_json` для tool-схем.

**DoD:** на тестовом SSH-таргете можно дать задачу формулировкой на естественном
языке (напр. «найди, какой процесс слушает порт 8080, и покажи его рабочую
директорию»), и агент пошагово, **с подтверждениями**, выполняет нужные команды и
выдаёт итог. Без подтверждения ни одна команда не выполняется. Лимит итераций
срабатывает на заведомо неразрешимой задаче.

> На этом этапе главная фича продукта уже существует и проверяема — дальше идёт UX.

## Этап 6 — TUI на ratatui (минимальный интерфейс)

**Цель:** удобный текстовый интерфейс поверх агентского ядра: чат + блоки команд.

**Задачи:**
1. Каркас `ratatui` + `crossterm`: область диалога, поле ввода, статус-бар
   (текущий таргет, режим confirm).
2. Рендер **блоков**: сообщение пользователя, рассуждение/ответ агента, блок
   команды (команда + её stdout/exit code), визуально разделённые.
3. UI подтверждения команды: `[a]pprove / [d]eny / [e]dit`.
4. Стриминг вывода выполняющейся команды в блок в реальном времени.
5. Скроллбэк, прокрутка, базовые горячие клавиши; переключение таргета
   (local ↔ ssh-host).

**Крейты:** `ratatui`, `crossterm`.

**DoD:** весь сценарий из Этапа 5 проходится **полностью из TUI**: ввод задачи,
наблюдение за блоками, подтверждение команд, чтение результата. Длинный вывод
скроллится, выполнение можно прервать клавишей.

## Этап 7 — Интерактивный терминальный режим (не агент)

**Цель:** дать пользователю обычный интерактивный терминал (как в любом эмуляторе),
рядом с агентским режимом. Здесь появляется реальная модель экрана терминала.

**Задачи:**
1. Подключить **готовую** модель терминала вместо собственной: `vte` (парсер ANSI)
   + `alacritty_terminal` (модель грида со скроллбэком, обработка escape-кодов,
   wide-символы, reflow при ресайзе). Своё с нуля **не писать**.
2. Прокинуть в эту модель поток из PTY (local) и из SSH-канала (remote).
3. Корректный проброс ввода (включая спецклавиши), ресайз окна → resize PTY/канала.
4. (Опц., zero-install-совместимо) Эфемерная **shell integration**: инъекция хуков
   (`PROMPT_COMMAND` для bash / `precmd` для zsh), которые эмитят OSC-маркеры начала/
   конца команды, exit code и cwd → это даёт «блоки» как в Warp и в интерактивном
   режиме. Помнить: это шелл-зависимо (bash/zsh/fish — разные реализации) и может
   конфликтовать с кастомным промптом пользователя; делать опциональным.

**Крейты:** `vte`, `alacritty_terminal`, `portable-pty`.

**DoD:** в интерактивном режиме корректно работают `vim`/`htop`/`less` (полноэкранные
TUI-приложения) и локально, и по SSH; ресайз не ломает экран. При включённой
shell-интеграции команды визуально группируются в блоки.

## Этап 8 — Полировка и (опционально) GPU-рендер

**Цель:** довести продукт; при желании — графический интерфейс уровня Warp.

**Задачи (по приоритету, опционально):**
1. История сессий, поиск по блокам, копирование вывода блока, шаринг команды.
2. Профили SSH-таргетов, быстрое переключение, мультиплекс нескольких сессий.
3. Графический фронтенд: миграция с `ratatui` на GUI — `wgpu` + текстовый рендер
   (`glyphon`/`cosmic-text`), либо фреймворк (`iced`/`gpui`). Это отдельный большой
   кусок; делать только когда ядро и UX устаканились.
4. Настройки темы, шрифтов, биндингов.

**DoD:** субъективно — продукт приятно использовать ежедневно; основные операции
(агент-задача, интерактивная работа, переключение хостов) выполняются без трения.

## Сквозные требования (держать на каждом этапе)

- **Безопасность секретов:** ключи только из env/секрет-стораджа, никогда в логах.
- **Тесты:** каждый этап закрывается интеграционными тестами; SSH — против Docker `sshd`.
- **Изоляция за трейтами:** `CommandExecutor` и `LlmClient` — точки замены реализаций.
- **Zero-install инвариант:** любой код, касающийся удалёнки, не должен оставлять
  файлов на диске удалённой машины; добавить тест-страж, проверяющий это.
- **Никаких команд без подтверждения** в режиме по умолчанию.
